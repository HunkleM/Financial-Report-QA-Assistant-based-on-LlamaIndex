# 02 评估指标与消融实验体系 (Evaluation & Metrics)

本文档定义了本项目核心的评估方法论（LLM-as-a-Judge）、测试集构建规范以及性能实验记录模板。所有评估均基于 M3 Ultra 的本地算力闭环执行。

---

## 1. 评估框架：本地化 RAGAS 闭环

本项目不依赖 OpenAI API，采用本地高隐私评估体系。

### 1.1 裁判模型配置 (Judge Setup)
在 `src/evaluation/ragas_eval.py` 中，必须将强大的本地模型实例化为 RAGAS 裁判：
- **Judge Model**: `Qwen2.5-72B` (经由 Ollama 本地部署，必要时允许降级至 7B + 人工核查)。
- **Judge Embedding**: `BAAI/bge-m3` (与系统中用于检索的嵌入模型保持一致)。

### 1.2 核心量化指标
RAGAS 将针对系统在测试集上的每一条回答，产出以下核心指标得分 (0.0 - 1.0)：
1. **Context Precision (上下文精确率)**: 衡量系统召回的 Top-K 文档块中，真正包含答案相关信息的块是否排在最前面。
2. **Faithfulness (幻觉率/忠实度)**: 衡量大模型最终生成的回答，是否 100% 由召回的文档块（Context）推导得出，有无凭空捏造金融数据。
3. **Answer Relevancy (回答相关性)**: 衡量回答是否直接命中了用户的原始问题。

## 2. 标准化测试集规范 (The 20-Question Golden Set)

为确保不同阶段（Phase 1 vs Phase 2）以及不同分块策略之间的评分具备横向可比性，系统必须固定一套包含 **20 个问题的金融测试题库** (`data/test_set.json`)。

测试集需涵盖以下四大维度（各 5 题），以全面考察系统的“细粒度抽取”与“宏观总结”能力：

1. **事实与数据提取 (Factoid Q&A)**:
   - *示例*: "2023 财年，公司的研发投入总额是多少？"
   - *考察点*: 测试 Fixed-256 小块召回和精准定位能力。
2. **因果与逻辑分析 (Reasoning)**:
   - *示例*: "报告中提到公司 Q3 利润率下降的主要原因有哪些？"
   - *考察点*: 测试上下文连贯性（语义分块优势区）。
3. **宏观总结与归纳 (Summarization)**:
   - *示例*: "请总结这份 50 页研报中提到的所有潜在政策风险。"
   - *考察点*: 测试跨文档理解（RAPTOR 树状摘要优势区）。
4. **结构化产物生成 (Artifact Generation)**:
   - *示例*: "请对比报告中 A 公司和 B 公司的市场份额，输出为表格格式。"
   - *考察点*: 测试大模型按照特定指令格式化输出长上下文的能力。

## 4. 交互可验证性补充评估（P2 必做）

除 RAGAS 指标外，建议在 Phase 2 报告中补充以下“产品可用性”数据：

- **引用点击跳页成功率**：目标 >= 95%
- **引用页码命中率（人工抽检）**：目标 >= 90%
- **伪造引用拦截率**：目标 >= 95%（无对应 source/page 时触发降级）
